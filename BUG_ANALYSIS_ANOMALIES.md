# 🐛 БАГ АНАЛИЗ: Несовпадение счётчиков needs_review (41 vs 17)

> **Статус:** ✅ ИСПРАВЛЕНО — BUG-037 (2026-05-01).  
> Счётчик в Обзоре теперь фильтрует только события за сегодня. Лимит аналитики увеличен до 1000.  
> Этот файл оставлен как исторический анализ.

## Проблема

На дашборде два несовпадающих счётчика "needs_review":
- **Обзор (01.05.2026):** "ТРЕБУЮТ ПРОВЕРКИ" = **41**
- **Аналитика (май 2026):** "АНОМАЛИЙ" = **17**

Пользователь в замешательстве: почему разные числа?

---

## 🔍 КОРЕНЬ ПРОБЛЕМЫ

### Обзор (Overview) — `loadOverview()`

**Строка 1356:**
```javascript
const [employees, todayEvents, reviewAll] = await Promise.all([
  guardedFetch('/rest/v1/employees?select=id,display_name,team,position&deleted_at=is.null&order=display_name.asc'),
  guardedFetch(`/rest/v1/events?...&photo_timestamp=gte.${todayStartUTC}&photo_timestamp=lte.${todayEndUTC}...`),  // ← ТОЛЬКО СЕГОДНЯ
  guardedFetch('/rest/v1/events?select=id&status=eq.needs_review'),  // ← БЕЗ ФИЛЬТРА ПО ВРЕМЕНИ!
]);
```

**Строка 1373:**
```javascript
document.getElementById('stat-review').textContent = reviewArr.length;  // = 41
```

**Что это значит:**
- Запрос за строке 1356 **НЕ имеет фильтра `photo_timestamp`**
- Получает **ВСЕ needs_review события от начала времён**
- Показывает **41** (ALL-TIME счёт)

guardedFetch(`/rest/v1/events?...

- Показывает **17** (только май, первые 200)

## 🎯 ВАРИАНТЫ ИСПРАВЛЕНИЯ

### ✅ **ВАРИАНТ 1: ПРАВИЛЬНЫЙ ПОДХОД** (Рекомендуемый)

Сделать оба счётчика согласованными:

**Вариант 1A: Оба показывают TODAY только**
```javascript
// Обзор (1356):
guardedFetch(`/rest/v1/events?select=id&status=eq.needs_review
  &photo_timestamp=gte.${todayStartUTC}     // ← ДОБАВИТЬ
  &photo_timestamp=lte.${todayEndUTC}`),    // ← ДОБАВИТЬ

// Аналитика остаётся как есть (показывает месяц)
```
**Результат:**
- Обзор покажет: 41 (needs_review сегодня)
- Аналитика покажет: 17 (needs_review в май)
- **Логично:** разные периоды → разные числа ✅

---

**Вариант 1B: Оба показывают ALL-TIME**
```javascript
// Аналитика (2046):
guardedFetch(`/rest/v1/events?select=id,employee_id,photo_timestamp,fraud_flags,name_from_photo
  &status=eq.needs_review
  // УБРАТЬ photo_timestamp фильтры!
  &order=photo_timestamp.desc
  &limit=500`),  // ← УВЕЛИЧИТЬ лимит
```
**Результат:**
- Обзор покажет: 41 (needs_review всех времён)
- Аналитика покажет: 41 (needs_review всех времён)
- **Логично:** но лишает аналитику контекста периода ❌

## 🔥 ДОПОЛНИТЕЛЬНЫЕ ПРОБЛЕМЫ

### Проблема 1: Лимит 200 в аналитике

**Строка 2046:**
```javascript
&limit=200  // ← Если needs_review > 200, их не видно!
```

**Риск:**
- Если в мае > 200 needs_review событий, счёт неправильный
- Сейчас: 17 < 200, но может вырасти

**Решение:** Удалить лимит или увеличить до 1000+
```javascript
&limit=1000  // или не указывать (по умолчанию 1000)
```

---

### Проблема 2: Логическое несовпадение

**Обзор:**
- Показывает только **СЕГОДНЯШНИЕ события** в таблице (line 1355)
- Но показывает **ВСЕ needs_review** в счётчике (line 1356)
- **Несогласованность:** таблица за день, счётчик за всё время

**Решение:**
- Либо счётчик за день
- Либо таблица за всё время

## 🚀 ФИНАЛЬНАЯ РЕКОМЕНДАЦИЯ

**ВЫБРАТЬ ВАРИАНТ 1A:**

```javascript
// dashboard/index.html, строка 1356

// БЫЛО:
guardedFetch('/rest/v1/events?select=id&status=eq.needs_review'),

// СТАТЬ:
guardedFetch(`/rest/v1/events?select=id&status=eq.needs_review
  &photo_timestamp=gte.${todayStartUTC}
  &photo_timestamp=lte.${todayEndUTC}`),
```

**Почему:**
1. ✅ Исправляет логику (одинаковые фильтры = одинаковые результаты)
2. ✅ Обзор показывает СЕГОДНЯШНИЕ проблемы (что важнее)
3. ✅ Аналитика показывает МЕСЯЧНЫЙ тренд (полезнее)
4. ✅ Пользователь видит разные числа в разных контекстах = нормально

Плюс удалить лимит 200 в аналитике (safety).
