### BUG-001: Vercel timeout убивает OCR — события теряются навсегда

- **Файл:** (отсутствует) `/vercel.json`
- **Приоритет:** 🔴 КРИТИЧЕСКИЙ
- **Статус:** ✅ Исправлено 2026-05-08
- **Описание:** OCR занимает 20–30 сек, Hobby-план Vercel = 10 сек. Функция убивается до `insertEvent()`, Telegram получил `200 OK` → событие потеряно **безвозвратно**.
- **Решение:** Создать `/vercel.json`:
```json
{
  "functions": {
    "api/webhook.js": { "maxDuration": 60 },
    "api/sign-photo.js": { "maxDuration": 10 }
  }
}
```

### BUG-002: CHECK constraint не включает 'duplicate' — бесконечный цикл

- **Файл:** `supabase/events.sql:10`
- **Приоритет:** 🔴 КРИТИЧЕСКИЙ
- **Статус:** ✅ Исправлено 2026-05-08
- **Описание:** AI-worker пытается сохранить `status='duplicate'` → PostgreSQL отклоняет → событие застревает в `processing` 15 мин → восстанавливается в `pending` → снова ошибка → **бесконечный цикл**.
- **Решение (Supabase SQL Editor):**
```sql
ALTER TABLE public.events
  DROP CONSTRAINT IF EXISTS events_status_check,
  ADD CONSTRAINT events_status_check
    CHECK (status IN ('pending','processing','done','needs_review','duplicate','rejected'));
```

### BUG-037: Token в localStorage — XSS уязвив при наличии XSS

### BUG-064: AI Worker обрабатывает события в порядке newest-first — departure раньше arrival

- **Приоритет:** 🔴 Критический
- **Статус:** ⏳ Ожидает исправления
- **Описание:** Запрос к pending событиям использует `order=created_at.desc`. Рабочий отправил приход в 08:00 и уход в 17:00. AI Worker берёт батч newest-first: уход (17:00) обрабатывается первым → departure без arrival → `is_duplicate=True` → `status=duplicate`. Затем приход (08:00) обрабатывается → departure не найден → `hours=null`. Итог: оба события испорчены, часы не считаются. Может затрагивать до 50% событий при каждом запуске воркера.
- **Решение:** Изменить `order=created_at.desc` → `order=created_at.asc`. Одна буква.

### BUG-065: Статус duplicate не имеет механизма восстановления

- **Файл:** `ai-worker/process_events.py`, строка ~920
- **Приоритет:** 🔴 Критический
- **Статус:** ⏳ Ожидает исправления
- **Описание:** `restore_stuck_events()` сбрасывает только `processing` → `pending` (после 15 мин). Для `duplicate` нет аналогичного механизма. События ставшие `duplicate` из-за BUG-064 (ложные дубли) застревают навсегда — ни воркер, ни ручная проверка их не трогает. Требует ручного SQL-сброса.
- **Решение:** После исправления BUG-064 выполнить SQL: сбросить все `duplicate` с `created_at >= 2026-05-01` где `fraud_flags` не содержит `"duplicate"` (настоящий дубль) → `pending`.

### BUG-066: withoutEnlargement: true блокирует upscale мелких штампов

- **Файл:** `scripts/ocr-worker.js` (Pass A resize)
- **Приоритет:** 🟠 Высокий
- **Статус:** ⏳ Устраняется заменой на EasyOCR (BUG-067)
- **Описание:** Sharp resize с `withoutEnlargement: true` не увеличивает изображение если оно уже меньше целевого размера. Telegram сжимает фото до 1280px → водяной знак занимает ~10px высоты. Tesseract требует минимум 12-15px для надёжного распознавания. Флаг блокирует upscale → Tesseract получает мелкий штамп → провал. Актуально для Pass A (полное фото).
- **Решение:** Убрать `withoutEnlargement: true`. Либо, предпочтительно, заменить Tesseract на EasyOCR (см. BUG-067).

### BUG-067: Tesseract не приспособлен для водяных знаков смартфонов — 62% провалов

### BUG-068: cleanup_photos.py удаляет фото без проверки photo_timestamp

- **Файл:** `ai-worker/cleanup_photos.py`

### BUG-069: waitUntil() не гарантирует завершение до убийства Vercel

- **Файл:** `api/webhook.js`
- **Приоритет:** 🟠 Высокий
- **Статус:** ✅ Принято / Accepted Risk — 2026-06-13
- **Описание:** `waitUntil()` позволяет Vercel завершить HTTP-ответ (200 OK Telegram) и затем продолжить async-обработку. Но Vercel не гарантирует что задача завершится — функция может быть убита до `insertEvent()`. Telegram уже получил 200 → повторов не будет → событие теряется. Ранее документировалось как BUG-001 (OCR-timeout). После удаления OCR вероятность снизилась: webhook теперь только EXIF + сжатие + insert (~2-3 сек) — значительно меньше 60-секундного лимита.
- **Решение:** Текущий риск приемлем. Мониторить Vercel Logs на `Timeout Error`. При повторении — убрать `waitUntil()` и делать insert синхронно (до return 200).

### BUG-071: ocr_worker.py — isoformat() в URL даёт 400 от Supabase

### BUG-075: Отрицательные часы при неверной паре arrival/departure

- **Файл:** `ai-worker/process_events.py`, строка 633
- **Приоритет:** 🟠 Высокий
- **Статус:** ✅ Исправлено — 2026-06-15
- **Описание:** При ошибке OCR (неверное время) или неправильном сопоставлении пары departure timestamp мог оказаться раньше arrival → `hours < 0`. Отрицательные часы попадали в табель и запутывали итоги.
