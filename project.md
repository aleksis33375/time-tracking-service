# project.md — AI Attendance System

---

## Что это за проект

AI Attendance System — сервис автоматического учёта рабочего времени сотрудников через распознавание лиц на фото. Бот добавляется в уже существующую Telegram-группу объекта, где сотрудники кидают фото при входе и выходе. Агент сам распознаёт лица, фиксирует время и формирует готовый табель — без ручной работы.

---

## Для кого

**Клиент / ниша:** строительная компания среднего масштаба (генподрядчик).

- 5 активных объектов
- ~120 рабочих и ИТР
- Руководители объектов вручную ведут учёт рабочего времени

---

## Проблема

- Учёт ведётся в Excel и на бумаге → постоянные ошибки
- Дубли сотрудников и «мёртвые души» в табелях
- Руководитель тратит 1–2 часа в день на сверку
- Конфликты с бухгалтерией и перерасход ФОТ
- Нет прозрачности: кто реально был на объекте и сколько работал

---

## Решение

Внедрили сервис учёта по фото:

- Сотрудники фиксируются через фото при входе/выходе
- Агент автоматически распознаёт лица и отмечает время
- Убирает дубли, формирует готовый табель
- Руководитель получает итог без ручной проверки

### Поток данных:
```
Фото в Telegram-группе → бот → Supabase Edge Function → AI Worker → запись в БД → Dashboard
```

---

## Результат

| Метрика | До | После |
|---|---|---|
| Время на учёт | ~1,5 часа/день | 10–15 минут/день |
| Ошибки и дубли | Постоянно | Практически нулевые |
| Оптимизация ФОТ | — | −8–12% |
| Закрытие месяца | 3 дня | Несколько часов |
| Прозрачность присутствия | Отсутствует | Полная |

---

## Ценность для клиента

- **Руководитель** освобождает время и фокусируется на стройке, а не на бумажках
- **Собственник** получает контроль без лишних проверок
- **Бухгалтерия** работает с точными данными, без конфликтов
- **Компания** снижает потери денег и управляет персоналом на основе фактов

> Вы не просто автоматизируете табель — вы убираете целый слой ручной, раздражающей и дорогой работы.

## Архитектура системы

### 1. Telegram Bot (Python)
- Участник существующей рабочей группы (не отдельный интерфейс)
- Ловит фото, отправленные сотрудниками
- Privacy mode отключён

### 2. Supabase (бэкенд)
- **PostgreSQL:** пользователи, фото-события, отметки времени
- **Storage:** хранение фото из Telegram
- **Edge Functions:** приём событий от бота → запуск AI-обработки

### 3. AI Worker (Python)
- Распознавание лиц (Face Recognition) на фото
- Идентификация сотрудника → фиксация времени прихода/ухода
- Исключение дублей

### 4. Vercel Dashboard (Frontend)
- Отчёты и таблицы посещаемости
- Просмотр фото
- Аналитика по объектам

## Формат фото и извлечение данных

Сотрудники снимают селфи через приложение **Timestamp Camera** (штамп даты/времени/локации) и добавляют подпись снизу — имя + тип события.

### Пример фото

```
┌─────────────────────────────────────────┐
│                    16 апр. 2026 г.,     │ ← автоштамп
│                    08:00:25             │
│                    Москва               │
│                    Россия               │
│         СЕЛФИ      108818               │
│        СОТРУДНИКА                       │
│                                         │
│                                         │
│         [ Дима начало смены ]           │ ← подпись вручную
└─────────────────────────────────────────┘
```

### Что система извлекает

**Сверху справа (OCR автоштампа):**
- Дата и время — когда сделано фото
- Город, страна — информационно
- Индекс — **идентификатор объекта** (108818 = наш объект)

**Снизу (OCR ручной подписи):**
- Имя — кто отметился (как он себя называет)
- Тип события — «начало смены» (приход) или «конец смены» (уход)

**Середина (Face Recognition):**
- Лицо сотрудника — используется как **верификация**: «на фото действительно тот, чьё имя указано в подписи?»

### Защита от подделок

- **Проверка индекса:** если в штампе не `108818` → фото с другого места → `needs_review`
- **Проверка лица:** если подпись «Дима», а лицо другого человека → возможный фрод → `needs_review`
- **Проверка времени:** несовпадения штампа и времени загрузки в Telegram → подозрение на старое фото

## Главный инструмент — страница «Табель»

### Роль
Основное рабочее место руководителя. Здесь формируется итоговый табель за период, отсюда данные уходят в бухгалтерию.

### Что видит руководитель на экране
Структура таблицы идентична привычному Excel:
- Слева — список всех сотрудников (ФИО, роль, ставка, ст-ть часа, переход часов)
- Сверху — дни периода
- В ячейках — отработанные часы или отметки «был / не был»
- Группировка по бригадам с зелёными подитогами
- Выходные подсвечены оранжевым
- Итоговые столбцы — часы и сумма

### Живая таблица (в отличие от Excel)
- Навёл мышку на день → всплыло фото сотрудника из Telegram
- Нераспознанные отметки подсвечены красным — сразу видно, где нужна проверка
- Поиск по ФИО прямо в таблице
- Бригады можно свернуть
- Выбор периода: день / неделя / месяц / произвольные даты

### Экспорт
Две кнопки:
- **Скачать Excel** — точная копия текущего файла для бухгалтерии (структура, листы, форматирование)
- **Скачать PDF** — для печати и подписи

Файл формируется прямо в браузере и скачивается.

### Как работает процесс
1. Руководитель открывает страницу «Табель», выбирает период
2. Dashboard подтягивает данные из Supabase
3. Формируется готовая таблица на экране (часы уже заполнены из распознавания фото)
4. Руководитель проверяет, при необходимости правит
5. Жмёт «Скачать Excel» → отправляет файл бухгалтеру по почте или мессенджеру

### Ключевое отличие от сегодняшнего дня
Часы в ячейках **заполняются автоматически** из фото через распознавание лиц. Руководитель не печатает — он **проверяет и подтверждает**.

### Принцип «одна роль»
Бухгалтерия **не заходит в систему**. Руководитель сам скачивает файл и передаёт — минимум сложности, никакого разграничения прав.

## Хранение данных и лимиты

### Где что хранится
- **`employees`** — сотрудники (имя, роль, бригада, ставка, эталонное фото, face embedding)
- **`events`** — отметки времени (кто, когда, приход/уход, часы)
- **`logs`** — журнал системных событий
- **Supabase Storage** — сами файлы фото из Telegram

### Как собирается таблица Табель
Таблица не хранится готовой. Каждый раз, когда руководитель открывает страницу «Табель», Dashboard запрашивает данные из Supabase (employees + events за период) и собирает таблицу в браузере на лету. Excel-файл формируется также в браузере при клике «Скачать».

### Лимиты Supabase Free Tier
- База данных: **500 МБ**
- Storage (фото): **1 ГБ**
- Трафик: **5 ГБ/мес**

### Ёмкость проекта (120 сотрудников)
- БД (тексты, события, логи): ~20 МБ/год → **хватит на 20+ лет**
- Storage без оптимизации: ~2,6 ГБ/мес → **не влезает в Free**

### Оптимизация Storage
- **Сжатие фото** при загрузке до ~150 КБ (качества достаточно для распознавания лиц)
- **Автоудаление** фото старше 30 дней
- Записи в БД (часы, даты, события) остаются навсегда — удаляются только файлы фото
- Активный объём: ~400–500 МБ → **вписывается в Free Tier**

### Когда может понадобиться Supabase Pro ($25/мес)
- Если клиент попросит хранить фото дольше 30 дней
- Pro даёт 100 ГБ Storage — хватит на 3+ года без сжатия

## Правила расчёта часов

Извлечены из Excel-файла заказчика (`Табель 23.03-08.04.xlsx`) через формулы и структуру ячеек.

### Базовые формулы

| Показатель | Формула | Пример |
|---|---|---|
| Норма рабочего дня | 8 часов | — |
| Стоимость часа | Ставка ÷ 8 | 5500 ÷ 8 = 687,5 |
| Часы за день | Время ухода − время прихода | 20:00 − 08:00 = 12 ч |
| Всего часов | Сумма часов за все дни периода | — |
| Сумма к выплате | Всего часов × Стоимость часа | 95 × 687,5 = 65 312,50 |

### Ключевые правила

- **Ставка = дневная** (не месячная). Оплата за полный 8-часовой день.
- **Обед не вычитается** — в ячейке полное время между первым «начало смены» и последним «конец смены».
- **«н»** в ячейке = не был на объекте, при суммировании считается как 0.
- **Выходные (сб, вс)** — рабочие дни на объекте, оплачиваются по той же ставке. Оранжевая подсветка — визуальная метка, на расчёт не влияет.
- **Дробные часы** допустимы (в файле встречается 12,5 часов).

### Переход часов

Между периодами сохраняется «переход часов» — сумма отработанных часов из предыдущего периода, копируется в первую колонку следующего:
- Первый период месяца (например, `23.03-31.03`) → перехода нет
- Следующий период (`01.04-08.04`) → в колонке «Переход часов» стоят итоги предыдущей недели

### Периоды и листы Excel

Экспорт формирует **отдельный лист Excel на каждый период**, обычно недельный:
- `23.03-31.03`
- `01.04-08.04`
- и т.д.

Это обязательно для совместимости с текущим процессом бухгалтерии.

### Бригады

Сотрудники группируются по бригадам. Между бригадами — **пустая строка-разделитель**, под каждой бригадой — **подитог суммой по бригаде** (зелёная подсветка). Имена бригад в файле не указаны, только структура.

## Чего не будет

- **Системы оплаты / расчёт зарплаты** — система фиксирует факт присутствия, но не считает деньги. Расчёт ФОТ остаётся в бухгалтерии / 1С.
- **Интеграция с 1С / ERP** — экспорт табеля в Excel/PDF есть, но прямой API-интеграции с учётными системами не будет.
- **HR-функции** — найм, увольнение, отпуска, больничные, графики смен вне скоупа.
- **GPS-трекинг** — система работает через фото в Telegram, не через геолокацию.
- **Видеонаблюдение** — только фото, не видеопоток и не камеры на объекте.
- **Управление задачами / нарядами** — это не таск-трекер и не система заданий.
- **Биометрические терминалы / железо** — никакого оборудования, только телефон и Telegram.
- **Мобильное приложение** — на старте интерфейс только через Telegram + веб-дашборд. Отдельное мобильное приложение возможно в будущем по запросу заказчика.
- **Множественные объекты** — на старте один объект. Поддержка нескольких объектов и привязка фото к объекту — в будущих итерациях.
- **Временные зоны** — один объект, один часовой пояс. Поддержка мультизонности не нужна.
- **Роли пользователей** — только руководитель. Собственник и бухгалтерия как отдельные роли с разграничением прав — не в скоупе.
- **Согласия ФЗ-152** — сотрудники уже отправляют фото в Telegram-группу добровольно. Юридическое оформление согласий на биометрию — пока не требуется.
- **Миграции БД / версионирование схемы** — Supabase CLI и формализованные миграции не используются. Схема правится напрямую.
